# DAH 2026 쉬운 설명서

이 문서는 팀원이 빠르게 이해할 수 있도록 DAH Flawless 프로젝트를 쉬운 말로 정리한 문서다.

---

## 1. 우리가 만든 것은 무엇인가?

드론 관제 AI가 보는 정보를 Red가 오염시키고, Blue가 그 거짓 정보를 찾아내는 **가상 공방 시뮬레이션**이다.

진짜 드론, 진짜 통신망, 실제 무기체계를 건드리지 않는다. 컴퓨터 안에서 안전하게 `world`와 `blue_observed`라는 상태 값을 바꿔가며 실험한다.

핵심 문장:

```text
본 시스템은 단순 룰 기반 탐지기가 아니라, 공격 성공/탐지 피드백을 이용해 전략을 조정하는 adaptive Red-Blue multi-agent architecture이다.
```

---

## 2. 등장인물

| 이름 | 역할 | 쉽게 말하면 |
|---|---|---|
| Red Agent | 공격 선택과 observed 변조 | 거짓 정보를 흘리는 공격자 |
| Threat Detection Agent | 이상 징후 탐지 | "이 정보 이상한데?"를 찾는 탐지반 |
| Mission Monitor Agent | 임무 영향 판단 | 이 이상이 임무에 얼마나 위험한지 보는 임무반 |
| Defense Planner Agent | 방어 action 선택 | 관찰/재검증/격리/복구를 고르는 방어반 |
| Incident Report Agent | 사건 요약 | 사람이 읽을 수 있게 사건을 정리하는 보고반 |
| Scorer | 승패 판정 | 진짜 상태를 아는 심판 |

---

## 3. 가장 중요한 개념

### world

진짜 상태다. 실제 배터리, 모터 상태, 정상 명령 번호, 진짜 임무 우선순위가 들어 있다.

### blue_observed

Blue 관제 AI가 받는 관측값이다. Red가 공격하는 대상이다.

예시:

```text
world.uav.battery_percent = 20
blue_observed.telemetry.battery_percent = 82
```

진짜 배터리는 20%인데 관제 AI 화면에는 82%로 보이는 상황이다.

### 진실 분리

Blue는 `world`를 보면 안 된다. 정답을 보고 맞히면 방어 AI가 아니라 정답지 확인이 되기 때문이다.

```text
Red도 world를 보지 않는다.
Blue도 world를 보지 않는다.
Scorer만 world를 본다.
```

## 4. 공격 3종

| 공격 | 쉽게 설명 | 위험 |
|---|---|---|
| `PRIORITY_POISONING` | 중요한 정찰 구역 점수를 바꾼다 | AI가 엉뚱한 곳을 우선 정찰 |
| `TELEMETRY_FDI` | 배터리와 모터 상태를 거짓으로 보이게 한다 | 복귀/안전모드가 늦어짐 |
| `TIME_DESYNC_REPLAY` | 과거 명령을 최신처럼 보이게 한다 | 위험한 명령이 다시 적용됨 |

---

## 5. Blue가 잡는 방식

Blue는 공격 이름을 직접 맞히지 않는다. 대신 관측값 안에서 말이 안 되는 조합을 찾는다.

예시:

```text
배터리는 82%라고 하는데
배터리 소모율은 높고
모터 상태도 앞뒤가 안 맞는다
=> telemetry 조작 의심
```

이런 "깨지면 안 되는 규칙"을 불변식이라고 부른다.

---

## 6. adaptive하다는 뜻

이 프로젝트는 완전한 강화학습 AI는 아니다. 하지만 라운드마다 피드백을 받아 전략을 조정한다.

Red 쪽:

- 탐지된 공격은 weight가 낮아진다.
- 탐지되지 않은 공격은 weight가 높아진다.
- adaptive stealth 모드에서는 탐지된 공격을 더 조용한 tactic으로 바꾼다.
- `TELEMETRY_FDI`는 `probe_delta`를 조절하며 탐지 경계를 탐색한다.

Blue 쪽:

- threat confidence가 낮으면 바로 막지 않고 관찰/재검증한다.
- 같은 도메인에서 이상이 반복되면 domain trust가 낮아진다.
- confidence가 높거나 trust가 낮으면 격리/복구로 방어 강도를 올린다.

그래서 "고정 if문 묶음"보다는 **피드백 기반 적응형 공방 에이전트 구조**라고 설명하는 것이 맞다.

## 7. 실행 방법

기본 실행:

```bash
PYTHONPATH=src python3 -m dah_flawless.main --seed 42 --rounds 5
```

더 강한 공방 조건:

```bash
PYTHONPATH=src python3 -m dah_flawless.main \
  --seed 42 \
  --rounds 5 \
  --scenario degraded_start \
  --red-stealth adaptive
```

대시보드:

```bash
PYTHONPATH=src streamlit run streamlit_app.py
```

테스트:

```bash
PYTHONDONTWRITEBYTECODE=1 PYTHONPATH=src python3 -m unittest discover -s tests -v
```

## 8. 결과는 어디서 보나?

시뮬레이션을 실행하면 기본적으로 아래 로그가 생성된다.

```text
data/logs/round_logs.jsonl
data/logs/summary.json
```

대시보드에서는 아래 탭으로 확인한다.

| 탭 | 설명 |
|---|---|
| Overview | 전체 승패, 탐지율, 공격 분포 |
| Timeline | 라운드별 공격/방어 흐름 |
| Scorer/Admin Diff | world와 observed 차이 |
| Charts | 간단한 그래프 |
| Decision Logs | Red/Blue 판단 로그 |

특히 `Decision Logs`와 JSONL의 아래 필드가 중요하다.

```text
feedback
red_policy_state
blue_policy_state
decision_log
```

이 필드들이 "피드백으로 전략을 조정한다"는 증거다.

## 9. 우리가 안 하는 것

- 실제 드론 해킹
- 실제 통신망 침투
- 실제 암호 해독
- 실제 무기체계 제어
- 딥러닝/강화학습 모델 학습

이 프로젝트는 안전한 시뮬레이션이다.

## 10. 용어 사전

| 용어 | 뜻 |
|---|---|
| schema | 데이터 구조의 정해진 모양 |
| state | 현재 상태를 담은 데이터 |
| world | 진짜 상태 |
| observed | AI가 보는 상태 |
| redaction | Blue에게 넘기기 전에 world를 지우는 것 |
| invariant | 깨지면 안 되는 규칙 |
| threat | Blue가 감지한 위협 |
| confidence | 위협이라고 믿는 정도 |
| domain trust | telemetry/mission/command 도메인별 신뢰도 |
| defense action | Blue가 취하는 관찰/재검증/격리/복구 행동 |
| availability | 임무 가용성 |
| RED_ATTRITION | 방어 비용 때문에 임무가 고갈되어 Red가 이긴 상태 |
| seed | 같은 결과를 재현하기 위한 랜덤 고정값 |
| JSONL | 한 줄에 한 라운드씩 저장하는 로그 형식 |
| hash chain | 로그 변조를 탐지하기 위한 해시 연결 |
| Streamlit | 결과를 보는 웹 대시보드 |
